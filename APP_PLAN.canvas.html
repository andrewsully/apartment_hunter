What We're Building

A collaborative apartment hunting app for 3 roommates, scraping Boston listings and layering on shared decision-making tools — ratings, comments, personal commute stats, and an MBTA-aware map.

📋
Browse

Grid or list view of all scraped listings. Filter by price, beds, neighborhood, pet policy. Sort by rent, date, rating, distance.

👥
Collaborate

3 user accounts. Each person has their own rating, favorites, and comments on every apartment. See at a glance what each roommate thinks.

🗺️
Map

Interactive Leaflet map. Pins color-coded by category. Optional MBTA overlay showing Red, Green, Orange, Blue lines + stations.

📏
Personal Stats

Each user sets their work address. Auto-calc walking or T commute time per apartment. See your commute score on every card.

Auth Approach
  • Flask-Login — session-based, very simple
  • No email verification — 3 hardcoded users or created via admin
  • Password hashing with werkzeug (already in Flask)
  • Invite code — optional: one-time code to register a slot
  • Persistent sessions — "remember me" so you don't re-login
  • Admin flag on User model — you can edit boundary, trigger scrapes
📐
Per-User Data
  • UserRating table — user_id, apt_id, stars (1–5), category (yes/maybe/no), is_favorite
  • Comment table — user_id, apt_id, text, created_at
  • UserDestination table — user_id, label, address, lat/lng, mode (walk/T/bike/drive)
  • CommuteCache table — user_id, apt_id, dest_id, minutes (cached result)

MBTA Data Source
  • MBTA Open Data — api.mbta.com or static GeoJSON
  • Static download recommended — download the route shapes GeoJSON once, store locally in data/mbta_routes.geojson
  • Lines: Red, Green (B/C/D/E), Orange, Blue, Silver (optional)
  • Stations: Name, lat/lng, line(s), accessibility (ADA)
  • Leaflet GeoJSON layer — style each line by color, add circle markers for stations
  • Toggle stored in localStorage — remembers your last state
⏱️
Commute Calculation
  • Google Maps Distance Matrix API — most accurate, handles transit
  • OpenRouteService — free alternative for walking/cycling
  • Fallback: straight-line distance — simple haversine, no API key needed, less accurate
  • Cache results in CommuteCache table — only recompute when address changes
  • Batch calculate on profile save — pre-fill commute for all current listings

Tech Stack Recommendations

Your existing stack (Flask + SQLite + Leaflet) is a great foundation. The additions below are all lightweight and keep complexity low.

Backend
Flask (existing)
Keep it — simple, already wired up with SQLAlchemy and the scraper
Database
SQLite (existing)
Fine for 3 users. Migrate to Postgres later if needed
Auth
Flask-Login
Minimal, session-based, no JWT complexity. pip install flask-login
Maps
Leaflet.js (existing)
Already in the app. Add GeoJSON layers for MBTA routes
MBTA Data
Static GeoJSON from MBTA Open Data
Free, no key needed, download once. URL: https://github.com/mbta/gtfs-documentation
Commute API
Google Maps Distance Matrix OR OpenRouteService
Google is most accurate (esp. transit). ORS is free/open-source. Even haversine works as fallback for walking rough estimate
Geocoding
Nominatim (OpenStreetMap)
Free geocoding for work addresses. No key needed. pip install geopy
Frontend
Vanilla JS + Jinja2 (existing)
No React needed. Keep it simple. Use fetch() for API calls, no build toolchain
CSS
Custom dark CSS (existing)
Keep current dark theme. Add CSS variables for consistency. No framework needed
Images
Local static files (existing)
Already working. Keep scraped images in static/images/apartments/

Optional Upgrades (if project grows)

🔄
Real-time Updates

Add Flask-SocketIO or Server-Sent Events so when one roommate votes, others see it live without refreshing. Nice but not MVP.

📱
Mobile PWA

Add a manifest.json and service worker to make the site installable on phones. Useful when you're out touring apartments.

🗄️
Postgres (Later)

If SQLite starts to feel limiting (lots of concurrent writes), migrate to Postgres. SQLAlchemy makes this a config change.

Build Phases

A suggested order of implementation. Each phase is independently useful — don't need to finish all of them to have something valuable.

1
Phase 1 — Polish the Existing UI Start Here
Make the current single-user listing view look great before adding complexity. Dark theme, grid/list toggle, filter sidebar, sort, card design with image + vote buttons. This is the core of the app and everything else builds on it.
Redesign index.html with proper dark theme Grid / List / Map view toggle Filter sidebar (rent, beds, no-fee, pets, neighborhood) Sort bar (price, date, rating) Apartment card with hero image + vote buttons Apartment detail page (full gallery, all fields) MBTA overlay toggles on map
2
Phase 2 — User Accounts & Per-User Ratings
Add Flask-Login, create User and UserRating models. Each person logs in and their votes/ratings are stored separately. Add the group consensus indicators to cards. Migrate existing rank/list_category/notes to be per-user.
User model + Flask-Login Login / logout pages UserRating model (stars, category, favorite, notes) Per-user vote API endpoints Group consensus display on cards Personal favorites page
3
Phase 3 — Comments
Add the Comment model and a comment thread to the apartment detail page. This is how roommates discuss specific listings without needing a group chat.
Comment model Comment thread UI on detail page Comment count badge on listing cards POST /api/apartments/:id/comments endpoint
4
Phase 4 — Commute Times & Personal Destinations
Each user sets their work address. The app geocodes it and calculates commute times to every apartment. Results are cached. Walk and T times are shown on cards and filterable in the sidebar.
